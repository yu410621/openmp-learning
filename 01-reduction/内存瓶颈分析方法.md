# 内存瓶颈系统性诊断方法论

> 以"OpenMP reduction 在 32 线程只有 10x 加速"为案例，
> 完整演示如何把"内存瓶颈"这个笼统结论拆解到具体层级。

---

## 0. 问题定义

```c
#pragma omp parallel for reduction(+:sum)
for (long i = 0; i < N; i++) {        // N = 1亿
    if (arr[i] > 0) sum += arr[i];
}
```

实测：i9-14900k（32 线程）加速比仅 10x，远低于理论 32x。

**要回答的问题不是"是不是内存瓶颈"（Roofline 已确认是），
而是"内存的哪个层级、哪种机制是瓶颈"。**

可能的内存瓶颈来源（需要逐个排除/确认）：

| 瓶颈类型 | 含义 | 典型场景 |
|---------|------|---------|
| L1 Bound | 数据不在 L1，等 L2 | 工作集略大于 L1 |
| L2 Bound | 等 L2/L3 | 工作集大于 L2 |
| **L3/LLC Bound** | **多核争抢共享 L3** | **多线程，工作集 > L2** |
| **DRAM Bandwidth Bound** | **内存总线带宽打满** | **大数组顺序访问** |
| DRAM Latency Bound | 等内存延迟（非带宽） | 随机访问、指针追逐 |
| NUMA Bound | 跨 socket 访问远端内存 | 多 socket 服务器 |
| TLB Bound | 页表查找开销 | 大内存、随机访问 |
| False Sharing | 多线程写同一 cache line | 共享变量、相邻写入 |

## 2. 第 1 层：Roofline 模型

### 2.1 计算 Arithmetic Intensity (AI)

```
循环每次迭代：
  - 读 1 个 long = 8 字节
  - 1 次比较 + 1 次加法 = 2 个操作

AI = 操作数 / 内存字节数 = 2 / 8 = 0.25 ops/byte
```

### 2.2 确定机器的 Roofline 拐点

```
i9-14900k:
  峰值算力（全核整数）≈ 数百 GOPS
  峰值内存带宽（DDR5-5600 双通道）≈ 89.6 GB/s（理论）

Roofline 拐点 = 峰值算力 / 峰值带宽
```

由于 AI = 0.25 远小于拐点（通常 10-30 ops/byte），**确定是内存带宽受限**。

### 2.3 实测机器带宽上限：STREAM

```bash
cd ~
git clone https://github.com/jeffhammond/STREAM.git
cd STREAM

# 编译，数组要足够大（远超 L3）
gcc -O3 -march=native -fopenmp \
    -DSTREAM_ARRAY_SIZE=100000000 \
    -mcmodel=large stream.c -o stream

./stream
```

关注输出的 Triad 和 Copy（单位 MB/s）：

```
Function    Best Rate MB/s
Copy:           XXXXX.X      ← 纯读写带宽
Triad:          XXXXX.X      ← a[i]=b[i]+c*d[i]，最常用基准
```

**记下这个数（假设 X GB/s），它是你 task1 的物理上限。**

### 2.4 计算 task1 实际消耗的带宽

```
task1 在 32 线程：时间 0.011 秒，数据 800 MB
实际带宽 = 800 MB / 0.011 s ≈ 72 GB/s
```

**对比 STREAM 实测值**：
- 如果接近 STREAM 上限 → 确认撞带宽墙
- 如果远低于 STREAM → 不是纯带宽问题，需要看延迟或其他

## 3. 第 2 层：Top-Down 微架构分析 (TMA)

TMA 把每个 CPU 周期归类到四大类，定位时间花在哪。

### 3.1 一级分类

```bash
perf stat -M TopdownL1 ./task1
```

输出四个百分比：

```
   X % tma_frontend_bound      取指/解码瓶颈
   X % tma_bad_speculation     分支预测失败
   X % tma_backend_bound       执行/内存瓶颈  ← 内存问题在这里
   X % tma_retiring            正常干活
```

**预期**：`backend_bound` 占大头（> 60%）。

### 3.2 二级分类（下钻 backend）

```bash
perf stat -M TopdownL2 ./task1
```

backend 进一步分：

```
   X % tma_core_bound          执行单元忙不过来（算力瓶颈）
   X % tma_memory_bound        等内存          ← 你的瓶颈
```

**预期**：`memory_bound` >> `core_bound`，证实是内存问题而非算力问题。

---

## 4. 第 3 层：内存层次下钻

这是回答"内存哪里有瓶颈"的核心步骤。

### 4.1 内存子分类

```bash
perf stat -M tma_memory_bound,tma_l1_bound,tma_l2_bound,tma_l3_bound,tma_dram_bound,tma_store_bound ./task1
```

输出会告诉你内存瓶颈具体在哪一层：

```
   X % tma_memory_bound
   X % tma_l1_bound       数据在 L1 但仍有延迟
   X % tma_l2_bound       等 L2
   X % tma_l3_bound       等 L3（多核争用常出现在这）
   X % tma_dram_bound     等主存          ← 大数组场景的主因
   X % tma_store_bound    写缓冲瓶颈
```

### 4.2 判断逻辑

```
如果 dram_bound 最大：
  → 数据太大，缓存放不下，主存带宽/延迟是瓶颈
  → 你的 task1（800MB >> 36MB L3）属于此类

如果 l3_bound 最大：
  → 多核争抢共享 L3
  → 减少线程数或改善数据局部性可能有帮助

如果 l1_bound / l2_bound 最大：
  → 工作集大小问题，考虑分块（tiling）

如果 store_bound 最大：
  → 写操作密集，考虑 non-temporal store
```

### 4.3 区分 DRAM 带宽 vs DRAM 延迟

`dram_bound` 还要进一步区分两种情况：

```bash
perf stat -M tma_mem_bandwidth,tma_mem_latency ./task1
```

```
   X % tma_mem_bandwidth   带宽打满（多线程顺序访问）  ← 你的场景
   X % tma_mem_latency     延迟受限（随机访问、指针追逐）
```

**这是关键区分**：
- **带宽受限**：多线程帮不上忙（总带宽固定），这就是你 10x 的根因
- **延迟受限**：增加并发可以隐藏延迟，多线程反而有效

## 5. 第 4 层：PMU 硬件事件精确量化

到这一层用具体硬件计数器，把假设变成数字。

### 5.1 核心事件采集

```bash
perf stat -e \
  cycles,instructions,\
  cache-references,cache-misses,\
  L1-dcache-loads,L1-dcache-load-misses,\
  LLC-loads,LLC-load-misses,\
  dTLB-loads,dTLB-load-misses,\
  branch-instructions,branch-misses \
  ./task1
```

### 5.2 关键指标计算

```
IPC (每周期指令数) = instructions / cycles
  健康：2-4
  内存受限：< 1

L1 D-cache 命中率 = 1 - (L1-dcache-load-misses / L1-dcache-loads)
  顺序访问应该 > 90%

LLC 命中率 = 1 - (LLC-load-misses / LLC-loads)
  大数组场景 < 50%（大量穿透到 DRAM）

分支预测准确率 = 1 - (branch-misses / branch-instructions)
  你的 if(arr[i]>0) 数据 4/7 为正，准确率可能偏低

dTLB miss 率 = dTLB-load-misses / dTLB-loads
  大内存可能偏高，触发 page walk
```

### 5.3 单线程 vs 多线程对比（最有信息量）

```bash
echo "=== 1 thread ==="
OMP_NUM_THREADS=1 perf stat -e cycles,instructions,LLC-loads,LLC-load-misses ./task1

echo "=== 32 threads ==="
OMP_NUM_THREADS=32 perf stat -e cycles,instructions,LLC-loads,LLC-load-misses ./task1
```

**预期观察**：

| 指标 | 1 线程 | 32 线程 | 解读 |
|------|--------|---------|------|
| total instructions | Y | ~Y | 总工作量不变 |
| total cycles | X | ~10X | CPU 总时间增 10 倍 |
| IPC | 高 | 降到 1/10 | 多线程被内存拖累 |
| LLC-load-misses | Z | ~30Z | 多线程争用 L3，miss 暴增 |

**LLC miss 在多线程下大幅增加 = 32 核共享 36MB L3，互相驱逐对方数据**。
这就是 USL 模型里 α（争用）项的微观证据。

## 6. 工具箱详解

### 6.1 perf（Linux 标准，必装）

```bash
sudo apt install linux-tools-generic linux-tools-`uname -r`
```

| 子命令 | 用途 |
|--------|------|
| `perf stat -M TopdownL1` | TMA 一级分类 |
| `perf stat -M TopdownL2` | TMA 二级分类 |
| `perf stat -e <events>` | 采集指定 PMU 事件 |
| `perf record` / `perf report` | 采样定位热点函数 |
| `perf mem record/report` | 内存访问剖析 |
| `perf c2c` | **检测 false sharing**（cache-to-cache） |

### 6.2 STREAM（带宽基准，必跑）

测量机器的峰值内存带宽，作为 Roofline 的上限参考。
GitHub: jeffhammond/STREAM

### 6.3 likwid（推荐，比 perf 友好）

```bash
sudo apt install likwid
# 或从源码编译 https://github.com/RRZE-HPC/likwid
```

```bash
# 测量内存带宽（运行时）
likwid-perfctr -C 0-31 -g MEM ./task1

# 测量 L3 缓存行为
likwid-perfctr -C 0-31 -g L3 ./task1

# 测量 cache 整体
likwid-perfctr -C 0-31 -g CACHE ./task1

# 微基准测试（类似 STREAM 但更细）
likwid-bench -t load -W N:1GB:32
```

likwid 的优势：**预定义了 MEM/L3/CACHE/FLOPS 等"性能组"**，
不用自己记一堆 PMU 事件名，输出还自动算好带宽、命中率。

### 6.4 Intel MLC（内存延迟检查器，免费）

```bash
# 从 Intel 官网下载 Intel Memory Latency Checker
# https://www.intel.com/content/www/us/en/download/736633/

./mlc                    # 全面测试
./mlc --bandwidth_matrix # 带宽矩阵
./mlc --latency_matrix   # 延迟矩阵
```

用途：**精确测量你机器在不同访问模式下的内存延迟和带宽**，
区分"带宽墙"和"延迟墙"的最权威工具。

### 6.5 Intel VTune（GUI，可选）

如果你想要图形化的内存分析：

```bash
# 下载 Intel oneAPI Base Toolkit 含 VTune
vtune -collect memory-access ./task1
vtune-gui  # 图形界面查看
```

VTune 的 "Memory Access" 分析直接给出每个内存层级的占比图，
适合不想记命令的场景。但对你（命令行老手）perf + likwid 足够。

## 7. 针对本案例的完整诊断流程

把上面方法论应用到你的 task1，一次跑完：

```bash
cd ~/openmp-learning/01-reduction
mkdir -p data

# === 第 1 层：带宽上限 ===
cd ~/STREAM && ./stream | tee ~/openmp-learning/01-reduction/data/stream.txt
cd ~/openmp-learning/01-reduction

# === 第 2 层：Top-Down 一级 + 二级 ===
perf stat -M TopdownL1 ./task1 2>&1 | tee data/tma-L1.txt
perf stat -M TopdownL2 ./task1 2>&1 | tee data/tma-L2.txt

# === 第 3 层：内存层次下钻 ===
perf stat -M tma_memory_bound,tma_l1_bound,tma_l2_bound,tma_l3_bound,tma_dram_bound \
  ./task1 2>&1 | tee data/tma-memory.txt

perf stat -M tma_mem_bandwidth,tma_mem_latency \
  ./task1 2>&1 | tee data/tma-bw-vs-lat.txt

# === 第 4 层：PMU 单线程 vs 多线程对比 ===
OMP_NUM_THREADS=1 perf stat -e cycles,instructions,LLC-loads,LLC-load-misses,cache-misses \
  ./task1 2>&1 | tee data/pmu-1thread.txt

OMP_NUM_THREADS=32 perf stat -e cycles,instructions,LLC-loads,LLC-load-misses,cache-misses \
  ./task1 2>&1 | tee data/pmu-32thread.txt
```

跑完后所有数据在 `data/` 目录，逐个分析。

## 8. 诊断决策树

遇到任何"并行程序不快"的问题，按这个树走：

```
程序加速比低于预期
│
├─ 算 AI，跑 Roofline
│  ├─ AI 高（计算密集）→ Core Bound → 优化算法/向量化
│  └─ AI 低（内存密集）→ 继续 ↓
│
├─ perf TMA 一级
│  ├─ Frontend Bound → 代码体积/指令缓存问题
│  ├─ Bad Speculation → 分支预测优化
│  ├─ Retiring 高 → 已经很好，考虑算法层
│  └─ Backend Bound → 继续 ↓
│
├─ perf TMA 二级
│  ├─ Core Bound → 执行单元瓶颈，向量化/减少依赖
│  └─ Memory Bound → 继续 ↓
│
├─ perf TMA 内存子分类
│  ├─ L1/L2 Bound → 工作集太大，分块(tiling)
│  ├─ L3 Bound → 多核争用，减线程/改局部性
│  ├─ DRAM Bandwidth → 带宽墙，多线程无解，需降低访存量
│  ├─ DRAM Latency → 延迟墙，增加并发/预取
│  └─ Store Bound → non-temporal store
│
└─ 特殊检查
   ├─ perf c2c → false sharing?
   ├─ numactl → NUMA 远端访问?
   └─ dTLB miss → 大页 (hugepage)?
```

## 9. 本案例的预期结论

对 task1（1亿 long 顺序求和）的诊断结论应该是：

```
根因排序（按贡献）：

1. DRAM 带宽墙（主因）
   - 800MB 数组 >> 36MB L3，几乎全部穿透到主存
   - 顺序访问，多线程同时拉满内存总线
   - 多线程不能增加总带宽 → 这是 10x 的核心原因

2. L3 cache 争用（次因）
   - 32 线程共享 36MB L3，互相驱逐数据
   - LLC miss 在多线程下显著增加

3. USL 一致性开销（小因）
   - 线程间 reduction 合并的同步开销
   - 量级较小，主要在 48 线程时才明显
```

**优化方向（如果要突破 10x）**：
- 降低访存量：用 `int32` 代替 `long`(int64)，数据量减半，带宽需求减半
- 改善数据布局：但顺序访问已经是最优局部性
- **本质上这个循环的 10x 就是物理极限**，除非改变算法的访存特征

## 10. 实战 checklist

诊断任何内存瓶颈问题时，确认完成了：

- [ ] 算出 arithmetic intensity，确认是否内存受限
- [ ] 跑 STREAM，知道机器带宽上限
- [ ] 算出程序实际带宽，对比 STREAM
- [ ] perf TopdownL1，确认 Backend Bound
- [ ] perf TopdownL2，确认 Memory Bound（非 Core Bound）
- [ ] perf 内存子分类，定位到 L1/L2/L3/DRAM 具体层级
- [ ] 区分 DRAM 带宽受限 vs 延迟受限
- [ ] 单线程 vs 多线程 PMU 对比，看 LLC miss 变化
- [ ] 检查 false sharing（perf c2c）
- [ ] 形成根因排序结论 + 优化方向

## 附录：术语速查

| 术语 | 含义 |
|------|------|
| AI (Arithmetic Intensity) | 算术强度，每字节内存的运算次数 |
| Roofline | 用算力和带宽上限判断瓶颈类型的模型 |
| TMA (Top-down Microarchitecture Analysis) | Intel 的流水线瓶颈分类方法 |
| IPC (Instructions Per Cycle) | 每周期指令数，越高越好 |
| LLC (Last Level Cache) | 最后一级缓存，通常是 L3 |
| PMU (Performance Monitoring Unit) | CPU 内置的硬件性能计数器 |
| False Sharing | 多线程写同一缓存行导致的伪共享 |
| NUMA | 非统一内存访问，多 socket 系统的内存拓扑 |
| TLB | 页表缓存，加速虚拟地址转换 |
| non-temporal store | 绕过缓存的写指令，减少缓存污染 |
